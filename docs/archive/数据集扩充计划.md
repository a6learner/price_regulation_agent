## 二、扩充策略（四种方式组合）

### 2.1 方式1：从133个PDF提取更多案例片段 ⭐⭐⭐

**优先级**: 最高
**预期产出**: 150-200个案例
**实施难度**: 中

#### 2.1.1 实施原理

现有133个PDF只提取了每份文书的主违规行为（1个案例/PDF），但实际上：
- 很多处罚文书包含**多个违规场景**（如：同时虚构原价+虚假折扣）
- 可以提取**不同商品**的违规行为（一个商家可能售卖多个违规商品）
- 可以分解为**多个独立案例**（如：同一商家在不同时间段的违规）

#### 2.1.2 实施步骤

**Step 1**: 创建多场景提取脚本 `scripts/expand_from_pdfs.py`

```python
核心功能：
1. 读取extracted_cases.jsonl（133个原始案例）
2. 使用LLM识别每份文书中的多个违规场景
3. 为每个场景生成独立的评估案例
4. 输出到data/eval/eval_from_pdfs.jsonl

提取策略：
- 识别多商品违规（如："商品A虚构原价，商品B虚假折扣"）
- 识别多时间段违规（如："2024年1-3月虚构原价，4-6月要素缺失"）
- 识别复合违规（如："虚构原价+虚假折扣"可拆分为2个案例）
```

**Step 2**: 人工审核与标注

```bash
# 生成初步案例
python scripts/expand_from_pdfs.py --input data/sft/processed/extracted_cases.jsonl \
                                   --output data/eval/eval_from_pdfs_raw.jsonl \
                                   --min-cases-per-pdf 1 \
                                   --max-cases-per-pdf 3

# 预期输出：133 PDFs × 平均1.5案例 = ~200案例

# 人工审核筛选
python scripts/review_expanded_cases.py --input data/eval/eval_from_pdfs_raw.jsonl \
                                        --output data/eval/eval_from_pdfs.jsonl \
                                        --quality-threshold 0.8
```

**预期产出**: 150-180个高质量案例

### 2.2 方式2：爬取新的处罚案例文书 ⭐⭐

**优先级**: 中
**预期产出**: 80-120个案例
**实施难度**: 高（需要爬虫开发）

#### 2.2.1 数据源

**目标网站**:
1. 国家市场监督管理总局 (https://www.samr.gov.cn/)
2. 各省市场监管局网站（安徽、江苏、浙江、广东等）
3. 信用中国 (https://www.creditchina.gov.cn/)

**爬取目标**: 2024-2025年电商价格违法处罚文书

#### 2.2.2 实施步骤

**Step 1**: 创建爬虫脚本 `src/data_collection/penalty_crawler.py`

```python
核心功能：
1. 爬取市场监管局网站的价格违法处罚公示
2. 下载PDF文书或提取HTML文本
3. 过滤出电商平台相关案例
4. 保存到data/cases/crawled/

爬虫策略：
- 关键词过滤："价格"+"电商"/"网络"/"平台"
- 时间范围：2024-01-01 至今
- 平台关键词：淘宝/京东/拼多多/美团/抖音等
```

**Step 2**: PDF提取与结构化

```bash
# 运行爬虫
python src/data_collection/penalty_crawler.py --start-date 2024-01-01 \
                                               --keywords "价格,电商,网络平台" \
                                               --output data/cases/crawled/

# 提取结构化数据
python scripts/数据处理/run_extract.py --input data/cases/crawled/ \
                                      --output data/eval/eval_from_crawler.jsonl
```

**预期产出**: 80-100个案例（假设爬取100-150份文书，提取率70%）

**风险与挑战**:
- 网站反爬虫机制
- PDF格式不一致
- 需要人工审核确保质量

### 2.3 方式3：基于现有案例生成变体 ⭐⭐⭐

**优先级**: 高
**预期产出**: 120-160个案例
**实施难度**: 低（成本较高但可控）

#### 2.3.1 生成策略

**三种变体类型**:

1. **参数变体**（简单改写）:
   - 修改：商品名称、价格数值、平台名称、时间
   - 保持：违规类型、违规逻辑、法律依据
   - 示例：`原价899元` → `原价1299元`

2. **场景变体**（中等改写）:
   - 修改：促销场景（限时折扣→满减活动→会员专享）
   - 修改：商品品类（服装→数码→食品）
   - 保持：违规类型和核心逻辑

3. **复杂度变体**（深度改写）:
   - Simple → Medium: 增加价格计算细节、多时间段对比
   - Medium → Complex: 增加多商品、多活动组合
   - 保持：违规类型判定标准

#### 2.3.2 实施步骤

**Step 1**: 创建变体生成脚本 `scripts/generate_variants.py`

```python
核心功能：
1. 读取eval_159.jsonl
2. 使用LLM生成变体案例
3. 自动标注meta字段
4. 质量验证（违规类型一致性检查）

Prompt策略：
- 参数变体：使用模板替换（不调用LLM，节省成本）
- 场景变体：使用Qwen3-8B生成（成本可控）
- 复杂度变体：使用Qwen3.5-397B生成（高质量）
```

**Step 2**: 批量生成与质量控制

```bash
# 生成参数变体（每个案例生成1个变体）
python scripts/generate_variants.py --input data/eval/eval_159.jsonl \
                                    --output data/eval/eval_variants_param.jsonl \
                                    --variant-type parameter \
                                    --variants-per-case 1

# 生成场景变体（针对特定类型）
python scripts/generate_variants.py --input data/eval/eval_159.jsonl \
                                    --output data/eval/eval_variants_scenario.jsonl \
                                    --variant-type scenario \
                                    --filter-types "要素缺失,虚假折扣" \
                                    --variants-per-case 2

# 生成复杂度变体（Simple→Medium）
python scripts/generate_variants.py --input data/eval/eval_159.jsonl \
                                    --output data/eval/eval_variants_complex.jsonl \
                                    --variant-type complexity \
                                    --filter-complexity simple \
                                    --target-complexity medium
```

**预期产出**:
- 参数变体: 159 × 1 = 159个
- 场景变体（针对要素缺失+虚假折扣）: 45 × 2 = 90个
- 复杂度变体: 88 simple → 30 medium

**总计**: ~160个案例

### 2.4 方式4：针对性补充特定类型 ⭐⭐⭐

**优先级**: 高
**预期产出**: 60-80个案例
**实施难度**: 中

#### 2.4.1 补充目标

根据Agent弱点分析，重点补充：

**1. 要素缺失案例（目标+60）**:
- 未标注原价（但标注折扣金额）
- 未标注运费/税费
- 未标注优惠条件（满减门槛、会员条件等）
- 未标注价格计算基准

**2. 虚假折扣案例（目标+65）**:
- 先涨后降（提价后打折）
- 折扣计算不实（标注8折实际9折）
- 虚假会员价（非会员也能购买）
- 比价欺诈（与其他平台对比虚假）

**3. 边界案例（目标+20）**:
- 价格误导 vs 要素缺失边界（强化区分）
- 虚假折扣 vs 虚构原价边界
- 价格误导 vs 虚构原价边界

#### 2.4.2 实施步骤

**Step 1**: 创建针对性生成脚本 `scripts/generate_targeted_cases.py`

```python
核心功能：
1. 定义每种违规类型的细分场景模板
2. 使用LLM基于模板生成案例
3. 确保覆盖Agent易错场景

模板示例（要素缺失）：
- 场景1：标注"立减500元"但未标明原价
- 场景2：标注折扣但未说明会员条件
- 场景3：标注"包邮"但未说明满减门槛
- ...（共15个细分场景）
```

**Step 2**: 批量生成

```bash
# 生成要素缺失案例
python scripts/generate_targeted_cases.py --type 要素缺失 \
                                          --count 60 \
                                          --output data/eval/eval_targeted_missing.jsonl

# 生成虚假折扣案例
python scripts/generate_targeted_cases.py --type 虚假折扣 \
                                          --count 65 \
                                          --output data/eval/eval_targeted_discount.jsonl

# 生成边界案例
python scripts/generate_targeted_cases.py --type 边界案例 \
                                          --count 20 \
                                          --output data/eval/eval_targeted_boundary.jsonl
```

**预期产出**: 60 + 65 + 20 = 145个案例（实际可能产出80-100个高质量案例）

## 三、数据整合与质量控制

### 3.1 数据整合流程

```bash
# 合并所有扩充数据
python scripts/merge_eval_datasets.py --inputs \
    data/eval/eval_159.jsonl \
    data/eval/eval_from_pdfs.jsonl \
    data/eval/eval_from_crawler.jsonl \
    data/eval/eval_variants_param.jsonl \
    data/eval/eval_variants_scenario.jsonl \
    data/eval/eval_variants_complex.jsonl \
    data/eval/eval_targeted_missing.jsonl \
    data/eval/eval_targeted_discount.jsonl \
    data/eval/eval_targeted_boundary.jsonl \
    --output data/eval/eval_500_raw.jsonl \
    --deduplicate

# 质量检查
python scripts/validate_eval_dataset.py --input data/eval/eval_500_raw.jsonl \
                                        --output data/eval/eval_500.jsonl \
                                        --quality-checks \
                                        --min-quality-score 0.7
```

### 3.2 质量控制标准

**必须满足的标准**:
1. ✓ 每个案例包含完整的messages字段（system + user + assistant）
2. ✓ meta字段包含7个必需字段（case_id, is_violation, violation_type等）
3. ✓ assistant回复包含：事实要点、合规分析、结论、法律依据、整改建议
4. ✓ case_id唯一且格式正确（eval_001 ~ eval_500）
5. ✓ 违规类型在预定义列表中（虚构原价/虚假折扣/价格误导/要素缺失/其他/不违规）

**建议满足的标准**:
- 每个违规类型至少60个案例（保证统计显著性）
- Simple:Medium:Complex ≈ 50:35:15（与原数据集比例接近）
- 平台分布相对均衡（每个平台≥40案例）
- 合规案例占比35-40%（避免数据不平衡）

### 3.3 去重策略

**三级去重**:
1. **精确去重**: case_description完全相同 → 保留1个
2. **语义去重**: embedding相似度>0.95 → 人工审核
3. **逻辑去重**: 违规类型+关键参数完全相同 → 保留1个
