# 8 System Implementation and Frontend Prototype

## 8.1 Overall System Architecture

The evaluation system is organized into three horizontal layers that communicate through well-defined interfaces. At the bottom sits the data layer, which provides the raw material for every experiment: the fixed evaluation set (`data/eval/eval_dataset_v4_final.jsonl`, 780 samples), a law knowledge base containing 691 statutory articles, and a case knowledge base of 133 historical administrative penalty records. These three repositories are logically separate — the evaluation set drives the test loop, the law KB feeds retrieval, and the case KB is held in reserve but deliberately excluded from formal scoring runs to prevent same-source contamination.

Above the data layer is the reasoning layer, where the three technical routes live side by side. The Baseline route sends each sample directly to Qwen-8B with a structured prompt and relies entirely on the model's parametric knowledge. The RAG route augments the same prompt with law articles retrieved at inference time through a hybrid retrieval pipeline. The Agent route replaces the single prompt call with a six-node workflow that sequences intent analysis, adaptive retrieval, relevance grading, chain-of-thought reasoning, heuristic reflection, and remediation advice generation. Despite the differences in reasoning orchestration, all three routes receive their input from the same data interface — a shared loader that reads the JSONL file line by line and yields a normalized sample dictionary — and they all hand their output to the same evaluation layer via a common result schema.

The evaluation layer sits at the top. It accepts the per-sample output from whichever reasoning route ran, computes the full metric set (binary accuracy, violation-type accuracy, precision, recall, F1, legal-basis quality, reasoning quality, evidence-chain completeness, legal-citation accuracy, remediation actionability, and interpretability), and writes the aggregated results as JSON. Because the output contract is shared, switching between routes requires nothing more than pointing the evaluation entry script at a different source module.

The technology stack is deliberately narrow. Python is the sole implementation language. LangChain provides the chain and agent abstractions, though in practice the Agent's linear node sequence is orchestrated directly through `AgentCoordinator` rather than a LangChain agent executor, giving us tighter control over retry logic and intermediate state. ChromaDB serves as the vector store backing both the law and case collections; it runs in embedded mode on the same machine, which keeps the setup dependency-free while still supporting cosine-distance queries. The model backend is Qwen-8B, accessed through a `maas_client` wrapper that handles API authentication via `configs/model_config.yaml` and exposes a uniform `complete()` interface to callers. BM25 lexical retrieval is handled in-process using a lightweight index built at vector-DB construction time, so no additional search service is required.

## 8.2 Backend Module Organization

### 8.2.1 Evaluation Entry Scripts

There is one entry script per technical route: `scripts/run_baseline_eval.py`, `scripts/run_rag_eval.py`, and `scripts/run_agent_eval.py`. Each script is responsible for loading the evaluation set, instantiating its own route-specific evaluator, iterating over samples, accumulating per-sample results, and writing the final JSON output. They accept command-line arguments — notably `--limit N` for smoke-test runs on a small slice of the dataset and, for the baseline script, `--models` to specify one or more model identifiers — but their internal logic is otherwise symmetrical.

All three scripts write results to a shared directory tree rooted at `results/`. The path convention is `results/{route}/{timestamp}/`, where `{route}` is one of `baseline`, `rag`, or `agent`, and `{timestamp}` is an ISO-formatted string stamped at the moment the run begins (e.g., `results/agent/20260418_113740/`). Inside each timestamped directory the script places a single JSON file containing every sample's input, predicted output, ground-truth label, and computed metric values, plus a summary block with aggregated statistics. This layout means that re-running an experiment creates a new subdirectory rather than overwriting an earlier one, making it straightforward to compare two runs of the same route under different settings. The three result files that underpin the numbers reported in this thesis are: `results/baseline/improved-1.md` (780 samples), `results/rag/improved-1.md` (780 samples), and `results/agent/improved_agent_full_eval-780__04-19/results.json` (780 samples).

### 8.2.2 RAG Module

The RAG module lives under `src/rag/` and consists primarily of two files. `retriever.py` implements `HybridRetriever`, which combines vector-similarity search and BM25 lexical search over the law collection. The two ranked lists are fused using Reciprocal Rank Fusion (RRF) before being passed to a CrossEncoder re-ranker that produces a final relevance score for each candidate article. After re-ranking, a three-stage dynamic filter is applied: a distance-threshold gate removes candidates whose cosine distance exceeds a configured ceiling; a minimum-k floor ensures that at least a small number of articles survive even when the threshold would discard everything; and a mean-distance-based dynamic Top-K cutoff trims the tail of the remaining list. When `cases_k` is set to zero — which is the fixed configuration for all formal evaluation runs — the retriever short-circuits the case collection query entirely, skipping both the ChromaDB lookup and the BM25 pass for cases. This avoids any possibility of the model being prompted with historical cases drawn from the same source as the test labels.

`evaluator.py` implements `RAGEvaluator`, which encapsulates the single-case evaluation flow: extract the query string from the sample, call `HybridRetriever` to get a ranked list of law articles (with `retrieval_info` returned for traceability), construct the augmented prompt, call the model, parse the response, and compute metrics. The separation between `retriever.py` and `evaluator.py` is deliberate: the Agent module reuses `HybridRetriever` directly rather than going through `RAGEvaluator`, which would have imposed unnecessary coupling to the RAG-specific prompt format.

### 8.2.3 Agent Module

The Agent module is rooted at `src/agents/`. The orchestrator is `agent_coordinator.py`, which implements `AgentCoordinator.process()`. This method drives the six nodes in a fixed linear sequence, passing a shared state dictionary from one node to the next. Each node populates specific keys in that dictionary; downstream nodes read from those keys without needing to know which upstream node wrote them.

The six node files and their roles are as follows. `intent_analyzer.py` is a pure rule-based module — it extracts query intent, named entities, and a complexity score from the penalty document text without making any LLM call, keeping latency low for this first step and fixing `cases_k` to zero unconditionally. `adaptive_retriever.py` wraps `HybridRetriever` and uses the complexity score produced by `IntentAnalyzer` to set the dynamic Top-K ceiling before retrieval; by reusing the same retriever class as the RAG module, the retrieval quality is consistent across routes. `grader.py` assigns a weighted quality score to each retrieved law article along three dimensions — relevance (weight 0.6), coverage (weight 0.3), and freshness (weight 0.1) — and filters out articles that fall below a minimum combined threshold, ensuring that only genuinely useful context reaches the reasoning step. `reasoning_engine.py` is where the main LLM call happens: it injects the filtered law articles into a system prompt that instructs the model to produce a structured JSON output following a five-step chain of thought (fact extraction → data verification → law matching → case reference → conclusion). `reflector.py` performs a zero-cost heuristic validation of the JSON output — checking for structural completeness, internal consistency, and confidence thresholds — and triggers at most one re-reasoning call to `ReasoningEngine` if a serious problem is detected; in the large majority of cases no retry is needed, which keeps average latency manageable. `remediation_advisor.py` reads the confirmed reasoning output and generates actionable remediation suggestions targeted at the specific violation type identified, producing the content that populates the `remediation_actionability` metric. Remediation output is architecturally absent from the Baseline and RAG routes; only the Agent route produces these suggestions.

### 8.2.4 Baseline Module

The Baseline module under `src/baseline/` has two files. `evaluator.py` handles the evaluation loop for a single sample: it builds a plain prompt containing only the penalty document text (no retrieved context), calls the model, delegates parsing to `ResponseParser`, and returns the metric dictionary. `response_parser.py` implements the heuristic scoring functions shared by all three routes — legal-basis quality, reasoning quality, and the binary classification extraction logic. Because these parsers are imported by the RAG and Agent evaluators as well, the quality metrics are computed consistently regardless of route, making cross-route comparisons valid on a like-for-like basis.

### 8.2.5 Knowledge-Base Construction Scripts

Before the RAG or Agent routes can be run for the first time, the ChromaDB vector store must be populated. Three scripts under `scripts/rag/` handle this in sequence. `rag_extract_laws.py` reads the raw law source files and outputs a normalized JSONL of 691 law articles, each with article text, article number, statute name, and effective date. `rag_process_cases.py` does the same for the 133 historical cases, extracting the case summary, violation type, and penalty outcome into a parallel JSONL file. `rag_build_vector_db.py` reads both processed files, encodes each entry using the configured embedding model, and writes the resulting vectors plus metadata into ChromaDB's two collections (`laws` and `cases`). BM25 index files are serialized alongside the ChromaDB data directory at this stage. The entire build process needs to run only once per knowledge-base version; subsequent evaluation runs load the pre-built index from disk.

All per-sample and aggregated results are saved as JSON rather than CSV or plain text. This format preserves nested structures — for instance, the full retrieval info dictionary or the multi-key Agent node output — without lossy flattening, and it is directly readable by the post-hoc analysis and visualization scripts used to generate the tables and figures reported elsewhere in this thesis.

## 8.3 Frontend Interaction Prototype (Proposed Design)

### 8.3.1 Design Goals

The backend evaluation pipeline described above is a batch research tool: it reads samples from a file, processes them in sequence, and writes JSON. A practitioner in a price-compliance supervision office cannot use it directly. The proposed frontend transforms the same Agent pipeline into an interactive single-document review tool, designed around four concrete user needs.

The first need is document ingestion: the user should be able to upload a PDF or image of an administrative penalty decision and have the system automatically extract structured fields — merchant name, penalty date, alleged violation type, cited statutory basis, and penalty amount — into an editable preview form. The second need is transparent reasoning: rather than receiving a single verdict at the end, the user should see each Agent node's output appear progressively, so that the decision process is legible rather than opaque. The third need is traceable evidence: every law article the Agent cites should be linkable back to its full text, reducing the cognitive load of cross-referencing paper copies of regulations. The fourth need is actionable follow-through: the remediation suggestions produced by `RemediationAdvisor` should be presented as an accept-or-revise card, lowering the barrier to acting on the system's output.

These goals translate directly into interface requirements: streaming output, a sidebar for cited-law display, structured-field display with inline editing, and a remediation card with confirmation controls.

### 8.3.2 Technology Selection

The proposed stack for the frontend prototype has three components.

Streamlit is the choice for the client-side UI layer. It allows a Python developer to build an interactive web interface without writing JavaScript, which matters here because the entire codebase is Python and adding a separate frontend framework would introduce a second language and a more complex build pipeline. Streamlit's `st.empty()` placeholder and `st.write_stream()` API support incremental text rendering, which is what the streaming agent output requires.

FastAPI wraps the Agent pipeline as an HTTP service. A thin adapter layer converts `AgentCoordinator.process()` into a POST endpoint at `/evaluate`, accepting a JSON body with the extracted document fields and returning a streaming response. FastAPI's native support for async generators makes it straightforward to yield each node's output as it completes rather than buffering the entire response.

Server-Sent Events (SSE) carry the streaming data from the FastAPI service to the Streamlit client. Each SSE message carries a JSON-encoded event payload with two fields: `node` (one of the six agent node names) and `content` (the text or structured output that node produced). The Streamlit frontend maintains a mapping from node name to a dedicated display container, and it updates each container as the corresponding SSE event arrives. This node-by-node streaming makes the Agent's six-step reasoning visible in real time rather than presenting a single block of text at the end.

### 8.3.3 Page Layout

The proposed interface is a three-pane layout that fills the browser window horizontally.

The left pane is narrow — approximately one quarter of the viewport width — and contains two stacked elements: a file upload widget at the top, and below it a structured-field preview card. After the user uploads a document and the backend OCR and field-extraction step completes, the preview card populates with labeled fields for merchant name, document date, violation type, cited articles, and penalty amount. Each field is editable inline so the user can correct extraction errors before triggering evaluation.

The center pane is the main working area and occupies roughly half the viewport width. It is organized as a vertical sequence of labeled node output blocks, one per Agent node, that appear progressively as SSE events arrive. Before the evaluation run begins, this area shows placeholder text for each node. As the run proceeds, each placeholder is replaced by the actual output of the corresponding node in sequence: first the intent summary from `IntentAnalyzer`, then the retrieval summary from `AdaptiveRetriever` (number of articles retrieved and their top relevance scores), then the grader's filtered list, then the full five-step chain-of-thought from `ReasoningEngine`, then the reflector's validation note (which in most cases simply confirms the output is consistent), and finally the remediation suggestion card from `RemediationAdvisor`. Below the node stream, a verdict banner displays the binary classification result (Violation / Compliant) and the violation type if applicable.

The right pane, roughly one quarter of the viewport width, is the evidence sidebar. It lists each law article that the Agent cited, with the article number and statute name as a clickable heading. Clicking a heading expands an accordion panel showing the full article text. For articles that have an effective date in the metadata, the date is displayed below the article text with a note if the article postdates the penalty document being reviewed.

### 8.3.4 Interaction Flow

The intended user journey through the interface is as follows. The user opens the application in a browser — served locally or on an internal network — and drags a penalty decision PDF onto the upload widget in the left pane. The backend's OCR and extraction step runs, taking a few seconds, and the structured-field preview card fills in. The user glances over the extracted fields, corrects the violation type if the OCR misread it, and then clicks the "Evaluate" button at the bottom of the left pane.

This action sends a POST request to the FastAPI `/evaluate` endpoint with the (possibly corrected) structured fields as the request body. The endpoint instantiates `AgentCoordinator`, begins processing, and opens an SSE stream. The Streamlit client connects to that stream and starts rendering node output into the center pane as events arrive. The user watches the reasoning unfold: first the intent classification appears, then the list of retrieved articles populates the right sidebar with clickable headings, then the graded subset is highlighted, then the chain-of-thought reasoning fills the largest block in the center pane.

If the user wants to verify a statutory citation, they click its heading in the right sidebar without interrupting the ongoing stream. The full article text expands in the accordion panel, and the user can read it while the remaining nodes continue to render in the center pane. When the remediation card appears at the bottom of the center pane, the user can click "Accept" to log the suggestion as-is or "Revise" to open an inline text editor and modify the suggestion before saving it to the case record.

### 8.3.5 Extensibility

Two extensions are planned for the follow-on development phase. The first is a batch mode that accepts a directory of penalty documents rather than a single upload. In batch mode the interface replaces the single-document layout with a progress table showing one row per document; each row displays the document filename, processing status, verdict, and a link to the detailed reasoning view. This mode is aimed at monitoring scenarios where a compliance officer needs to triage a day's worth of newly published penalty decisions efficiently.

The second extension is an audit-log view. Every evaluation run already produces a timestamped JSON file under `results/agent/`. The audit-log page reads these files and presents them in a searchable table indexed by document identifier, evaluation timestamp, and verdict. A compliance officer can navigate to any past evaluation, view the full node-by-node reasoning stream as it was at the time of the run, and compare it against any manual review notes saved alongside the result. This view is particularly important for accountability: because the underlying knowledge base and model version may change over time, the audit log preserves a snapshot of the exact reasoning that supported each historical decision.

## 8.4 Reproducibility and Engineering Discipline

Reproducibility is a practical concern rather than a formal property claim. The evaluation set is fixed: `eval_dataset_v4_final.jsonl` was written to disk and never modified after it was finalized for these experiments. The file contains 780 samples covering five violation categories and the compliant class, and we treat it as read-only throughout all reported runs. Any future experiment that modifies the evaluation set will create a new versioned file (following the `_v5_` naming convention established by earlier iterations).

Model decoding is configured for determinism. The API calls to Qwen-8B set temperature to zero and top-p to one, which in practice yields deterministic output for a fixed input on the inference backend we use. We acknowledge that strict mathematical determinism cannot be guaranteed across model-backend updates, but in the runs we conducted we observed no variation between repeated calls on the same sample.

Each run is stamped with a wall-clock timestamp at the moment `AgentCoordinator` or the corresponding evaluator is initialized. That timestamp propagates into the result directory name (e.g., `results/agent/20260418_113740/`) and into a metadata block inside the JSON output file, which also records the law-KB version identifier (a hash of the ChromaDB collection snapshot taken at build time) and the model identifier string. This means that any single numeric result cited in this thesis — for instance, the Agent's reasoning-quality average of 0.8931 — can be traced back to a specific JSON file in a specific timestamped directory (e.g., `results/agent/improved_agent_full_eval-780__04-19/results.json`), and from there to the exact KB snapshot and model configuration that produced it.

The knowledge-base version is pinned at construction time. The ChromaDB data directory is not regenerated between runs unless the source law files change. If the law articles are updated — for example, because a new price-regulation notice is published — the build scripts must be re-run and the new KB version must be explicitly tagged, ensuring that evaluation results computed against different KB snapshots are never inadvertently compared.

## 8.5 Limitations of the Current Implementation

Three limitations are worth stating explicitly.

The frontend described in Section 8.3 has not been implemented. It is a design that emerges from the backend's existing output structure — the node-by-node SSE scheme maps directly onto `AgentCoordinator`'s sequential processing, and the structured-field schema is already present in the evaluation JSONL — but the Streamlit and FastAPI code does not yet exist. Building it is planned for the follow-on phase; the design is documented here so that an implementer has a clear enough specification to begin without revisiting the architecture.

The evaluation pipeline runs on a single machine with no parallelism across samples. For the 780-sample dataset, the Agent route takes approximately 8.1 hours wall-clock time (37.62 seconds per sample on average), which is manageable for a one-off research experiment but would become a bottleneck in production monitoring where thousands of new penalty documents might arrive per week. Distributing the evaluation loop across workers — for instance, using a task queue with one worker per available GPU — is a straightforward engineering step that we have not yet taken because the single-machine setup was sufficient for the experiments reported here.

Knowledge-base maintenance is entirely manual. When new price-regulation statutes are issued or existing articles are amended, someone must manually update the source law files, re-run `rag_extract_laws.py` and `rag_build_vector_db.py`, and re-tag the new KB version. There is no automated subscription to regulatory publication feeds or continuous synchronization with the source document repository at cfws.samr.gov.cn. In a production deployment this gap would be significant: a compliance system that silently operates against a stale law KB could miss recently enacted rules or continue citing articles that have been superseded. Addressing this would require a scheduled crawler or webhook integration with the regulator's publication system, which is beyond the scope of the current research prototype.
