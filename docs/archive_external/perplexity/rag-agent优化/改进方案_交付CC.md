# 价格合规系统改进方案（交付 CC 执行）

> **文档目的**：向执行方（CC）交付当前系统的问题诊断与改进思路。本文不含具体代码实现，仅交代**问题在哪里 / 改什么 / 为什么这样改 / 预期效果**，具体代码由 CC 完成。
>
> **定位调整（重要）**：本轮改进的核心目标是**提升二分类准确率与决策可信度**；指标关注点从"违规类型分类正确率"转移到**法律逻辑严谨性、检索质量、证据可追溯性**等体现 Agent 与 RAG 真正价值的维度。类型分类仍保留为参考指标，但不再作为优化主轴。

## 一、现状与核心问题（基于三份 misclassified 文件的量化分析）

### 1.1 误分类方向高度一致

| 系统 | 总错 | FP（误报：合规→违规） | FN（漏报：违规→合规） | FP 中"误导性价格标示"占比 |
|---|---|---|---|---|
| Baseline | 79 | **51** | 28 | 33/51 = 65% |

## 二、改进方案（按优先级与依赖关系排序）

### P0-1 清理 Agent 中残留的案例依赖（对应用户问题③）

**目标**：彻底移除"案例参考"这个 prompt 层面的幽灵步骤，让 Agent 在 `cases_k=0` 时输出和决策都不再涉及案例。

**改动点**：

1. **`reasoning_engine.py` — 条件化 CoT 模板**
   - 把当前硬编的 5 步模板改为**条件化生成**：根据 `graded_docs['graded_cases']` 是否非空，动态决定是 4 步还是 5 步。
   - 4 步链（`cases_k=0` 时默认）：事实提取 → 价格要素校验 → 法律条款匹配 → 得出结论。
   - 只有检索确实命中了案例（`cases_k>0` 且 `len(graded_cases)>0`）才追加"参考相似案例"步骤。
   - 在 system prompt 中显式加入**禁令**：
     > "禁止引用上下文未提供的历史案例或处罚判例。若【相似处罚案例】为空，不得在推理链、legal_basis 或 reasoning 中出现"类似案例""历史案例""处罚判例"等表述。"

2. **`reasoning_engine.py` — 去除空 cases 的填充**
   - 当 `cases` 为空时，**整块不进入 prompt**（连"【相似处罚案例】（暂无相似案例）"这行都不要出现），避免任何暗示。

3. **`grader.py` — 短路 cases 处理**
   - 当输入 `retrieved['cases']` 为空时，跳过评分和过滤，`graded_cases` 直接置空；`filtering_stats` 里也不再输出 `cases_before/after`（或显式标记 `cases_disabled: true`）。

4. **`intent_analyzer.py` — 去除案例相关提示**
   - `reasoning_hints` 中不再包含"参考 XX 平台相关案例"（line 198）这类语句。

**预期效果**：Agent 推理链从 91/91 都带"步骤4: 参考相似案例"降到 0；彻底消除这部分噪声输出和 token 浪费。

### P0-2 引入"风险提示"中间态输出（对应用户问题②）

**目标**：解决"合规样本中存在灰色瑕疵"导致的 FP 问题。当前系统只能输出 `is_violation=true/false` 二元结果，面对 COMP_B_002（"全网最低价"无依据）、COMP_B_010（平台间原价不一致）等样本，系统识别出的风险是真实存在的，但 GT 标为合规——这时应给出"合规但有风险"的中间态，而不是强行判违规。

**改动点**：

1. **三套 prompt（baseline / rag / agent）统一扩展输出 schema**：

   ```
   {
     "is_violation": false,              // 是否构成价格违法（严格的法律判定）
     "violation_type": "无违规",
     "has_risk_flag": true,              // 新增：存在合规风险但未构成违法
     "risk_level": "low/medium/high",    // 新增
     "risk_categories": ["促销信息披露不充分", "价格比较基准缺失"],  // 新增
     "risk_description": "...",          // 新增：具体说明风险点
     "risk_suggestions": [...],          // 新增：针对风险的改进建议
     "legal_basis": "...",
     "cited_articles": [...],
     "reasoning": "..."                  // baseline/rag
     "reasoning_chain": [...]            // agent
   }
   ```

2. **在 system prompt 中明确三种判定态**：
   - **违规（is_violation=true）**：案情事实完整，且明确违反具体法条的字面要求，存在误导后果或实质损害。
   - **合规无风险（is_violation=false, has_risk_flag=false）**：案情未涉及价格要素，或价格标示完整规范。
   - **合规但有风险（is_violation=false, has_risk_flag=true）**：存在灰色瑕疵（如"全网最低价"无比较基准、未标活动时间、信息披露不充分），尚未构成违法，但建议整改。

3. **只有 `has_violation=true` 时 `RemediationAdvisor` 才生成整改建议**；`has_risk_flag=true` 时生成"风险提示建议"（`risk_suggestions`），区分严重程度。

4. **评测侧新增指标**：
   - `Risk Flag Recall`：合规样本中被识别出真实瑕疵的比例（针对 `COMP_B_*`、`COMP_C_*` 这类本身有瑕疵的合规样本）。
   - `Over-enforcement Rate`：`FP / (FP+TN)` —— 直接度量保守问题。
   - 把"合规但有瑕疵"的样本从 FP 计数中**移出**（如果系统给出 `has_risk_flag=true` 就不计 FP）——这能让 Agent 的真实判别能力得到公平评估。

**预期效果**：47 个三系统共错 FP 中，预计 35+ 个会变成 `has_risk_flag=true` 的正确判定；Agent 的过度判违率从 24.6% 降到 10% 以下；Agent 的独特价值（细粒度风险识别）得以体现在指标中。

### P0-3 收紧"误导性价格标示"的判定边界（对应问题②的另一面）

**目标**：这是 FP 最集中的类型（Agent 50/69=72%）。根因是**当前 prompt 对该类型的定义过宽**，且 intent_analyzer 的 hint 机制存在"确认偏差"。

**改动点**：

1. **三套 prompt 统一加入"误导性价格标示"的判定四要件**：

   > 判定为"误导性价格标示"需同时满足以下四点，缺一不可：
   > - **(a) 存在明示的价格比较或优惠宣传**（原价/划线价/折扣率/"最低价"等）；
   > - **(b) 缺乏真实依据或客观基准**（例如：无前 7 日交易记录、无同期同商品对比、无发布机构可核查）；
   > - **(c) 足以使一般消费者产生错误价格认知**（存在具体误导后果，而非仅形式瑕疵）；
   > - **(d) 与"不明码标价""政府定价违规""标价外加价"等类型不重叠**。
   >
   > 仅存在下列情形的，**不判为违规，列为 `has_risk_flag=true`**：
   > - 促销活动未标注起止时间；
   > - 赠品规则 / 满减规则披露不充分；
   > - 不同平台标注的原价不一致，但未造成实际消费者选购误导；
   > - "全网最低价""保价双十一"等宣传语缺乏依据但未直接导致价格虚高结算。

2. **在 prompt 中增加"不构成误导性价格标示"的反例示范**（2~3 条）：
   - 例：实际结算金额**低于**标价（向下抹零、让利），**不属于**价格欺诈；
   - 例：标注原价 X、促销价 Y 且 X 有真实交易记录的，**不属于**误导；
   - 例：案情涉及商标侵权、产品质量但未涉及价格误导的，一律判"无违规（不属于价格合规范畴）"。

3. **`intent_analyzer.py` 收紧关键词规则**：
   - "误导性价格标示" hint 命中条件从"出现原价/促销等关键词"改为"**同时**出现'原价/划线价'且出现'从未成交/无交易记录/不符/虚构'"两侧的匹配。
   - `reasoning_hints` 改为**中性表述**：不再说"关注价格标注是否真实"（这是确认偏差），改为"若案情涉及价格比较，核查被比较价格的真实性与客观依据"。

**预期效果**：FP 中"误导性价格标示"数量显著下降（Agent 预计从 50 → 20 以内）；二分类准确率直接提升约 3~5 个百分点。

### P0-4 重构 Reflector 的校验维度（从"形式检查"转向"法律逻辑检查"）

**目标**：当前 Reflector 的启发式校验本质是**格式检查**（推理链长度、字段一致性），没有检查法律推理本身的质量。论文中 Agent 的价值应该体现在"反思能发现法律逻辑错误"，而不是"反思能发现输出字段不齐"。

**改动点**：

1. **废弃"推理链长度 < 3 即问题"**（line 103-111）——长度与正确性无关。

2. **新增三条有实际价值的校验规则**：

   - **校验 A：法条适用性匹配**
     检查 `cited_articles` 中的法条，是否与 `violation_type` 的法律类目匹配。例如：
     - `violation_type='不明码标价'` → 应命中《价格法》第十三条、《明码标价和禁止价格欺诈规定》第四~六条；
     - `violation_type='政府定价违规'` → 应命中《价格法》第十一~十二条；
     - 若引用了不对应的条款（如用《反不正当竞争法》判"不明码标价"），标记为 `severity=warning`。

   - **校验 B：事实要素完备性**
     根据 `violation_type`，检查推理链是否包含该类型所需的**构成要件事实**：
     - `不明码标价` 必须有"未标注/缺失品名/计价单位/规格/价格"中至少一项；
     - `政府定价违规` 必须有"政府指导价/政府定价"与"实际收费"的对比；
     - `标价外加价` 必须有"标价"与"实收"的金额差；
     - `误导性价格标示` 必须有四要件中至少前三项。
     若事实要素缺失，标记为 `severity=critical`，触发重新推理。

   - **校验 C：非价格领域排除**
     若推理链或 user query 核心事实属于"商标侵权/食品安全/产品质量/资质证照/税务"且无价格要素，但 `is_violation=true`，标记为 `severity=critical` —— 这条规则直接解决 CASE_0042/0044/0052 这类 FN 的对立面（避免同类问题反向出现在 FP 中）。

3. **Reflector 返回的 `issues_found` 写入最终结果**，作为"证据链完整性"高级指标的实打实来源，而不是当前这种"格式性检查"。

**预期效果**：Agent 的"推理质量"指标从"检查形式" → "检查法律逻辑"；Reflector 触发重推理能真正纠错，而非现在几乎不触发（因为格式校验容易通过）。

### P1-1 改造 RemediationAdvisor：从规则模板 → 基于事实的 LLM 生成

**目标**：当前 `mode="fast"` 使得所有违规案例输出**完全相同的 3 步模板**，失去可评估意义。

**改动点**：

1. **`agent_coordinator.py`** 把 `mode="fast"` 改为 `mode="llm"`（或根据复杂度动态选择）。
2. **`RemediationAdvisor` 的 prompt** 要求：
   - 输入：案情事实 + `violation_type` + 命中的具体法条 + 违规事实要素；
   - 输出：**针对本案具体事实**的整改动作（例如对 COMP_B_002 应输出"补充'全网最低价'的比较基准与数据来源，或将宣传语改为'本店特惠价'"），而非泛化模板；
   - 包含"法律风险评级 / 责任主体 / 整改时限 / 预期处罚区间"等字段，每一项都需绑定具体事实。
3. **新增指标：整改建议特异性（Remediation Specificity）** —— 用向量相似度或 LLM 打分评估"建议内容"与"案情事实"的相关性，`rule_based` 占比越低越好。

**预期效果**：`remediation.generation_mode` 从 100% `rule_based` 转向 90%+ `llm`；整改建议可操作性指标真正反映系统能力而非模板覆盖率。

### P1-2 Prompt 共性优化（三系统同步）

**baseline/rag/agent 三套 prompt 共同新增以下内容**（这是保证三系统横向可比的前提）：

1. **"案例事实不足"的退出规则**：
   > 若案情描述未提供与"价格标示/收费/结算/定价机制"相关的具体事实，一律输出 `is_violation=false, violation_type='无违规', confidence<=0.6, reasoning='案情未涉及价格要素，不属于价格合规范畴'`。

2. **"法条引用约束"**（重点针对 RAG）：
   > `cited_articles` 中的每一条法条，**必须**出现在本次提供的【相关法律条文】上下文中。禁止凭记忆引用未提供的法条编号。若上下文法条不足以支撑判定，应在 reasoning 中说明"可用法条不足"并降低 confidence。

3. **"类型分类不是首要目标"的提示**（与你本轮调整指标关注点一致）：
   > 优先保证 `is_violation` 判定的准确性与法律逻辑严谨性。若违规存在但类型难以精确归类，可归入"其他价格违法"并在 reasoning 中说明依据。

### 3.3 数据集整体建议

1. **引入"evidence_sufficiency"字段**：给每条样本标记 `sufficient / partial / insufficient`。`insufficient` 样本在主指标中剔除或单列。
2. **新增 `compliant_with_risk` 子类型**：专门用于评测"风险识别"能力（对应 P0-2 的 `has_risk_flag`）。
3. **剔除或隔离"非价格领域"样本**：CASE_0042/0044/0052/0056/0061 这类商标/食品/质量案例应不在价格合规评测集内。

## 四、评测指标体系调整（核心关注点转移）

### 4.1 原指标体系存在的问题

- **违规类型准确率（70% 左右）**被当作核心指标，但它把"合规类 / 违规类"两种判断绑定在一起：只要二分类对但子类型错（例如把"不明码标价"误分类为"其他价格违法"），就扣分。这让 Agent 的真实判别能力被类型标注噪声拖累。
- **法律引用准确性（最高 0.160）**：基于启发式字符串匹配，当前实现对 RAG/Agent 很不利——模型引用了正确的法条但条款号写法略有不同就被判错。
- **整改建议可操作性**：100% 靠 rule_based 模板拿到的分数，没有区分"真实建议"与"模板填空"。

### 4.2 建议的新指标权重

| 维度 | 原权重 | 建议权重 | 说明 |
|---|---|---|---|
| 二分类准确率 | 中 | **高（主指标）** | 本轮改进的首要目标 |
| 违规类型准确率 | **高** | 低（参考） | 不再作为优化主轴；在二分类正确前提下观察细粒度能力 |
| 法律依据质量 | 中 | **高** | 体现 Agent/RAG 相对 Baseline 的核心价值 |
| 法律引用准确性 | 低 | **高** | 需要重写校验逻辑，从字符串匹配改为"法条-违规类型"语义匹配 |
| 检索质量（新增） | — | **高** | 新增：`laws_recall@k`（是否命中正确法条）、`laws_precision@k`、过滤有效率 |
| 证据链完整性 | 中 | 高 | 继续保留，但基于 P0-4 的新 Reflector 校验结果重新计算 |
| 风险识别召回率（新增） | — | 中 | 针对 `has_risk_flag`，度量"合规但有瑕疵"的识别能力 |
| 过度判违率（新增） | — | 中 | FP / (FP+TN)，直接度量保守问题 |
| 整改建议特异性（新增） | — | 中 | 替代"可操作性"，要求非模板 |

### 4.3 建议的报告口径

论文/毕设报告中建议采用**三层叙事**：

1. **分类能力层**：Baseline/RAG/Agent 二分类准确率对比（本轮目标从 88~90% 提升到 93%+）。
2. **法律逻辑层**：法律引用准确性 + 证据链完整性 + 法条-类型匹配率，这是 Agent 相对 Baseline 的**核心增量价值**。
3. **可解释性与可操作性层**：风险识别召回率 + 整改建议特异性 + 结构化输出完整度，这是 Agent 相对 RAG 的**核心增量价值**。

这样既诚实地呈现了"分类能力三者相近"，又能讲清 Agent 的真实价值所在。

## 六、与 CC 协作的建议

1. **每完成一个 P0 项先跑小规模回归**（50 条验证集），确认方向对再扩展全量。
2. **不要同时改动多个文件的 prompt**——先按 P0-1 → P0-3 → P0-2 → P0-4 的顺序单独验证，再合并。
3. **保留改动前的 prompt 版本**（在 git 中打 tag），便于消融实验：论文的 ablation study 需要 "去掉案例步骤 / 加入风险提示 / 加入法律逻辑校验" 三个单独变体的对比数据。
4. **风险提示中间态上线后，必须同步改评测脚本**（`src/baseline/evaluator.py` 的指标统计函数）否则 `has_risk_flag=true` 的样本仍会被算作 FP。
5. **`CASE_0042/0044/0052` 这类标注错误样本在数据清洗完成前，建议先在评测脚本里加一个"豁免清单"跳过**，避免继续污染指标。
