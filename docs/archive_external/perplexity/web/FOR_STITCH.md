# 价格合规智能体 Web UI — 交给 Stitch 的设计需求

> Stitch 使用原则提醒:一次一个改动,不要把多需求塞一个 prompt;用 Web 模式 + Gemini 2.5 Pro Thinking;满意的版本立刻截图存档。每个页面开新对话。

## 产品一句话

面向**消费者 / 政府监管 / 网店商家**三类用户的价格合规 AI 助手,输入文字/文档/图片,输出带法条溯源的合规判定与整改建议。

## 五个页面

| 路由 | 名称 | 一句话功能 |
|---|---|---|
| `/` | 角色入口 | 三张卡片选身份(消费者/监管/商家) |
| `/chat` ⭐ | 对话工作台 | 聊天主界面 + 右侧溯源抽屉(核心页,80% 使用时间) |
| `/batch` | 批量审核 | 上传 Excel/多份 PDF,进度条 + 结果表 |
| `/knowledge` | 知识库 | 浏览 691 条法规 + 133 条案例 |
| `/compare` | 三路线对比 | 同一 query 并排展示 Baseline / RAG / Agent 差异 |

## `/chat` 必须有的元素(唯一强约束)

- 左侧:历史会话列表(可折叠)
- 中间:聊天流。输入框支持文本 + 附件按钮(文档/图片)
- AI 回复每条下方固定出现:**判定结论(合规/违规 + 置信度)**、**违规类型**、**法律依据徽章(可点)**、**整改建议**
- 六节点进度条(意图→检索→评分→推理→反思→整改),AI 思考时实时高亮
- 右侧可呼出**溯源抽屉**,展示:
  - 命中法条原文 + 具体条款号
  - 检索得分明细(向量/BM25/RRF/重排 四个分)
  - 相似历史案例卡片
  - 一键跳原始文书来源

## 三角色差异化(用同一套组件,不画三版)

预留"角色徽章位"和条件卡片槽,按角色决定哪些卡片默认展开:

| 卡片 | 消费者 | 监管 | 商家 |
|---|---|---|---|
| 判定结论 | 展开 | 展开 | 展开 |
| 法律依据 | 折叠 | 展开高亮 | 折叠 |
| 相似案例 | 隐藏 | 展开 | 折叠 |
| 整改建议 | 隐藏 | 折叠 | 展开高亮 |
| 维权建议 | 展开高亮 | 隐藏 | 隐藏 |
| 处罚建议 | 隐藏 | 展开高亮 | 隐藏 |

## 风格倾向(供 Stitch 自由发挥的底线)

- 气质:专业、政务友好、不花哨。对标:Linear 的清爽 + 政务系统的可信感
- 中文优先,字号略大(监管用户年龄偏大)
- 溯源是卖点,证据展示要有"法律文书感"(等宽字体、条款编号突出、可引用样式)
- 留白充足,避免密集表格

## 建议的 Stitch 工作顺序

1. **先定设计系统**:一次 prompt 只问"给一个政务/合规 AI 工具的设计系统:配色、字体、按钮、卡片样式",截图存
2. **做角色入口**(最简单,练手)
3. **做对话工作台主结构**(不含抽屉)
4. **做溯源抽屉**(单独一屏,最花时间,迭代 5 次以上不过分)
5. **做剩余三页**(批量/知识库/对比)
6. 全部导出 Figma,交给 Claude Code 还原成 React

## 具体提示词怎么写(官方建议)

- 主提示 ≤ 5000 字符,把"页面名 + 核心元素列表 + 风格关键词 + 参考产品"一次说清
- 修改时一条 prompt 只改一处:"Make the sidebar collapsible" 而不是 "Make sidebar collapsible and change button color and add search bar"
- 用 UI 术语:sidebar drawer, breadcrumb, card layout, empty state, skeleton loader
