# 合规样本构建指南 (Compliant/Non-Violation Sample Construction)

> 给 Claude Code 的执行指南
> 目标: 为 eval_dataset_v3_final.jsonl (500条违规) 补充 ~300 条合规样本
> 日期: 2026年4月

---

## 背景与约束

### 当前数据集
- 500 条违规案例，全部来自真实处罚文书 PDF
- 缺少合规(无违规)案例 → 无法评测 binary classification 的 false positive rate
- 需要补充 ~300 条合规案例，最终数据集 ~800 条

### 核心约束
1. **不能全用同一个 LLM 生成** — 否则 Baseline 用同源 LLM 就能轻易识别
2. **合规案例需要足够真实** — 不能是明显无关的文本，必须是"看起来像可能违规但实际合规"的 hard negatives
3. **需要多样性** — 覆盖不同行业、平台、价格行为类型
4. **格式与违规案例完全一致** — 共用同一套 schema

---

## 三源混合策略 (Three-Source Mix)

**核心思路: 300 条合规样本来自 3 个不同来源，避免单一来源的同质性问题。**

```
合规样本来源:
├── Source A: 真实合规场景改写 (~120条) ← 基于违规案例的"修正版"
├── Source B: 真实商业场景构造 (~120条) ← 基于真实商品/服务的合规定价描述
└── Source C: 边界/易混淆案例 (~60条)  ← hard negatives，接近违规但实际合规
```

---

## Source A: 违规案例修正版 (~120条)

### 原理
取现有违规案例，将违规要素修正为合规状态。这样生成的样本与违规案例在文本风格、行业背景上高度一致，只是核心违规事实被修正。

### 方法

从 500 条违规案例中，按违规类型分层采样 120 条作为模板:

```
不明码标价 (228条 → 抽55条)
政府定价违规 (120条 → 抽28条)
标价外加价 (73条 → 抽17条)
误导性价格标示 (49条 → 抽12条)
其他类型 (30条 → 抽8条)
```

### 修正规则（按违规类型）

#### 不明码标价 → 合规版: 已明码标价

```
原文(违规): "执法人员检查发现，该超市货架上摆放的7瓶洗手液未标明价格"
修正(合规): "执法人员检查发现，该超市货架上摆放的7瓶洗手液均已张贴标价签，
           标明了品名、产地、规格、计价单位及销售价格，标价内容清晰完整"
```

#### 政府定价违规 → 合规版: 执行政府指导价

```
原文(违规): "该停车场收费标准为小型车10元/小时，超出政府指导价8元/小时"
修正(合规): "该停车场收费标准为小型车8元/小时，符合市发改委核定的
           政府指导价标准，并在入口处醒目位置公示了收费依据文号"
```

#### 标价外加价 → 合规版: 无额外收费

```
原文(违规): "当事人在标价之外向消费者收取50元的'包装费'，未事先标明"
修正(合规): "当事人商品售价包含所有费用，未额外收取包装费等未标明的费用，
           商品标价签上注明了'含包装'字样"
```

#### 误导性价格标示 → 合规版: 价格标示真实

```
原文(违规): "商品页面标注'原价1680元，现价188元'，经查该商品从未以1680元销售过"
修正(合规): "商品页面标注'原价1680元，现价188元'，经查该商品在2024年1月至3月
           期间确有以1680元成交的交易记录共23单，原价标注真实"
```

### 执行步骤

1. 从 eval_dataset_v3_final.jsonl 按类型分层采样 120 条
2. 对每条，提取 case_description 中的关键违规要素
3. 用 LLM (Qwen3.5-397B 或 MiniMax-M2.5，**不用 Qwen3-8B**) 生成修正版
4. Prompt 模板见下方

### LLM Prompt (Source A)

```
你是一位中国市场监管领域的合规专家。

我会给你一段来自行政处罚决定书的违法事实描述。
请你将其修改为"合规版本"——即保留相同的商业场景（行业、商品/服务、平台），
但将违法要素修改为合规状态。

修改规则:
1. 保留: 当事人类型、行业、商品/服务类型、经营场所、时间段
2. 修改: 将违法行为改为合规行为（如"未标价"→"已按规定标价"）
3. 添加: 适当补充合规细节（如"标价签注明了品名、产地、规格、计价单位"）
4. 风格: 保持行政执法文书的客观叙事风格，不加入主观评价
5. 长度: 与原文相近（±30%）
6. 禁止: 不引用任何法律条文，不做法律分析结论

原文（违规）:
{original_case_description}

违规类型: {violation_type_name}

请输出修正后的合规版本（仅输出修正文本，不要其他说明）:
```

### Ground Truth

```jsonc
{
  "is_violation": false,
  "violation_type": null,
  "qualifying_articles": [],     // 合规案例无应引用法条
  "penalty_articles": [],
  "legal_analysis_reference": "",
  "penalty_result": ""
}
```

### 原理
基于真实的商业活动构造合规定价场景。这些不是从违规案例改来的，而是独立构造的合规商业描述。好处: 增加文本多样性，避免 Source A 可能存在的模式（修正痕迹）。

### LLM Prompt (Source B)

```

请你撰写一段关于以下商业场景的事实描述，描述内容应该体现该商户在价格方面的合规做法。

要求:
1. 风格: 模拟市场监管执法人员的日常检查记录，客观中性
2. 内容: 描述商户的定价/标价行为，体现其符合价格法规定
3. 必须包含: 具体的商品/服务名称、具体价格数字、标价方式
4. 长度: 150-400字
5. 禁止: 不引用法律条文，不做"合规"或"违规"的判断结论
6. 禁止: 不能出现"合规""符合规定""未违反"等判定词

场景类别: {scenario_category}
具体场景: {specific_scenario}
地区: {random_region}

请输出事实描述（仅输出描述文本）:
```

## Source C: 边界/易混淆案例 (~60条)

### 原理
构造 hard negatives: 看起来像违规但实际合规的场景。这类样本能有效检测系统的 false positive rate。如果模型只要看到"原价""折扣""标价"等关键词就判违规，hard negatives 会暴露这个问题。

### 6类边界场景（10条/类 = 60条）

#### C1. 合法使用"原价"标注 (10条)
```
场景: 商品确实有以原价交易的记录，折扣是真实的
关键: 与"虚构原价"相似，但有真实交易记录支撑
示例: "当事人在美团平台销售公猫绝育套餐，标注原价1680元，
      折扣价188元。经查，2024年1月至3月期间该套餐以1680元
      成交12单，原价标注有真实交易记录。"
```

#### C2. 合法价格差异（不同规格/服务等级）(10条)
```
场景: 同一商品在不同页面价格不同，但原因是规格/配置差异
关键: 与"首页/详情页不一致"相似，但价格差异有合理原因
示例: "当事人在拼多多平台销售手机壳，首页标示价格9.9元（透明软壳），
      详情页另有一款标价19.9元（磁吸防摔壳），两款为不同规格商品。"
```

#### C3. 合法浮动定价（市场调节价）(10条)
```
场景: 价格有波动但在合法范围内
关键: 与"哄抬价格"相似，但涨价幅度合理且属于市场调节价
示例: "当事人经营的蔬菜批发市场，白菜批发价从1.2元/斤涨至1.8元/斤，
      涨幅50%，经查系因连日暴雨导致产地供应减少，属正常市场波动。"
```

#### C4. 合法使用比较价格 (10条)
```
场景: 标注"市场参考价"并有数据来源
关键: 与"虚假价格比较"相似，但比较价格有依据
示例: "当事人在商品页面标注'市场参考价58元（数据来源：京东平台同款
      商品均价）'，当事人售价39元。经核实，京东平台同款商品近30日
      均价确为56-62元区间。"
```

#### C5. 促销活动规则执行正确 (10条)
```
场景: 促销活动描述复杂但实际执行一致
关键: 与"不履行价格承诺""促销规则不一致"相似，但规则被正确执行
```

#### C6. 标价中有附加条件但已明确标注 (10条)
```
场景: 有附加消费条件但在显著位置清晰标注了
关键: 与"隐藏不利价格条件"相似，但条件已充分公示
```

### LLM Prompt (Source C)

```
你是一位中国市场监管领域的合规专家。

请你构造一个"边界合规案例"——即表面上看起来可能涉嫌违规，但实际上是合规的价格行为。

边界类型: {boundary_type}
具体场景描述: {scenario_description}

要求:
1. 风格: 模拟市场监管部门现场检查的事实记录
2. 必须包含: 看似违规的要素（如提到"原价""折扣""价格差异"等），但随后给出合规的解释
3. 解释应该是事实性的（如"经查有真实交易记录"），而不是法律判断
4. 长度: 150-400字
5. 禁止: 不引用法律条文，不做最终"合规/违规"结论性判断
6. 禁止: 不出现"未违反""合规""不构成违法"等判定词

请输出事实描述:
```

## 质量控制 (Quality Control)

### Step 1: 自动检查

对所有 300 条合规样本运行以下自动检查:

```python
QUALITY_CHECKS = {
    # 1. 长度检查: 与违规案例分布相似
    "length_check": lambda desc: 80 <= len(desc) <= 600,
    
    # 2. 判定词泄露检查: 不能包含直接的合规/违规判断
    "no_judgment_words": lambda desc: not re.search(
        r'(合规|违规|不违法|未违反|符合规定|不构成违法|无违法行为)', desc
    ),
    
    # 3. 法条泄露检查: 不能引用法条
    "no_law_citation": lambda desc: not re.search(
        r'《.*?法》|《.*?规定》|第\w+条', desc
    ),
    
    # 4. 具体性检查: 必须包含具体的价格数字
    "has_price_info": lambda desc: bool(re.search(
        r'\d+(?:\.\d+)?元', desc
    )),
    
    # 5. 场景真实性: 必须包含具体的商品/服务/场所
    "has_entity": lambda desc: bool(re.search(
        r'[\u4e00-\u9fa5]{2,}(?:店|市场|平台|公司|超市|药店|医院|学校)', desc
    )),
}
```

### Step 2: LLM 交叉验证

用一个不同的 LLM（如果生成用的 Qwen3.5-397B，验证用 MiniMax-M2.5）检查:

```
请判断以下事实描述中是否存在价格违法行为:
{case_description}

仅回答: "违规" 或 "合规"
如果回答"违规"，简要说明违规要素。
```

如果验证 LLM 判断为"违规"，说明该合规样本确实是 hard negative (好事，保留)。
如果验证 LLM 判断为"合规"，正常保留。
如果验证 LLM 指出描述中存在自相矛盾或不合理的地方，则修正或丢弃。

### Step 3: 人工抽检

从 300 条中随机抽 30 条 (10%) 人工检查:
- 是否读起来像真实的行政检查记录？
- 事实描述是否自洽？
- 是否真的合规？(有没有写着写着变成违规的情况)

## 统一格式 (Output Schema)

合规样本的格式与违规样本完全一致:

```jsonc
{
  "id": "COMP_A_001",            // 编号规则: COMP_{来源}_{序号}
  "source_pdf": null,             // 合规样本无对应PDF
  "region": "广州市天河区",        // 随机分配或场景指定
  "tier": 0,                      // tier=0 表示合成样本
  "source_type": "compliant_corrected",  // 新增字段，标记来源
  // source_type 取值:
  //   "compliant_corrected" = Source A (违规修正版)
  //   "compliant_constructed" = Source B (真实场景构造)
  //   "compliant_hard_negative" = Source C (边界案例)

  "input": {
    "case_description": "...",    // 合规场景事实描述
    "platform": "美团",           // 涉及平台（可空）
    "goods_or_service": "公猫绝育套餐"  // 涉及商品/服务
  },

  "ground_truth": {
    "is_violation": false,        // ★ 核心: false
    "violation_type": null,
    "qualifying_articles": [],    // 空: 合规案例无应引用法条
    "penalty_articles": [],
    "legal_analysis_reference": "",
    "penalty_result": ""
  }
}
```

## LLM 使用策略 (避免同源污染)

### 关键原则
评测时 Baseline 使用 Qwen3-8B，所以合规样本生成**绝对不能用 Qwen3-8B**。

### 推荐方案

```
Source A (120条): Qwen3.5-397B 生成
Source B (120条): MiniMax-M2.5 生成
Source C (60条):  Qwen3.5-397B 和 MiniMax-M2.5 各30条

交叉验证: 用生成时没用到的那个模型验证
```

如果只有一个模型可用，至少确保:
1. 生成模型 ≠ Baseline 评测模型 (Qwen3-8B)
2. 用不同的 temperature (0.7-0.9) 增加多样性
3. 每条给不同的场景细节，避免模板化

## 执行步骤总结

```
Step 1: 准备工作
  ├── 从 eval_dataset_v3_final.jsonl 按类型分层采样 120 条 (Source A 模板)
  ├── 准备 12类×10场景 = 120 个场景描述 (Source B)
  └── 准备 6类×10条 = 60 个边界场景描述 (Source C)

Step 2: LLM 生成 (需API，约 300 次调用)
  ├── Source A: 用 Qwen3.5-397B 生成 120 条修正版
  ├── Source B: 用 MiniMax-M2.5 生成 120 条构造版
  └── Source C: 混合生成 60 条边界版

Step 3: 质量控制
  ├── 自动检查 (长度/泄露/具体性)
  ├── LLM 交叉验证
  └── 人工抽检 30 条

Step 4: 合并
  ├── 500 条违规 + 300 条合规 = 800 条
  ├── 输出 eval_dataset_v4_final.jsonl
  └── 更新 dataset_statistics.json
```

### 成功标准

```
✅ 合规样本 300 条
✅ 三种来源比例: A:B:C ≈ 40%:40%:20%
✅ 自动检查通过率 ≥ 95%
✅ LLM 交叉验证无自相矛盾 ≥ 90%
✅ 判定词泄露率 < 2%
✅ case_description 平均长度在违规案例的 ±30% 范围内
✅ 生成模型 ≠ Baseline 评测模型 (Qwen3-8B)
```

## 重要提醒

### ❌ 不要做的事
1. **不用 Qwen3-8B 生成合规样本** — Baseline 用 Qwen3-8B，会导致同源可识别
2. **不在 case_description 中写"该行为合规"** — 这是答案泄露
3. **不生成脱离实际的场景** — "某超市卖苹果价格合理"这种空洞描述没有评测价值
4. **不让所有合规案例都太"干净"** — 需要 Source C 这样的 hard negatives

### ✅ 必须做的事
1. **三源混合** — 不能全来自一种方法
2. **标记 source_type** — 后期分析时要看不同来源的评测差异
3. **保证长度分布自然** — 合规样本不能比违规样本明显短或长
