## 3. Claude Code之前方案的致命问题

### 3.1 同源污染（Source Contamination）— 最核心问题

**问题描述**：

```
测试集生成链路：
  原始PDF → 规则提取 → qwen-8b改写（统一模板格式）→ 测试集753条
  
Baseline系统链路：
  用户Query → qwen-8b → 回答
  
结果：
  测试集文本风格 = qwen-8b风格
  Baseline的输出 = qwen-8b风格
  → Baseline在"风格对齐"上有不公平优势
  → 评测指标（BLEU/ROUGE/embedding similarity）虚高
```

**量化风险**：如果测试集中所有reference answer都经过qwen-8b改写，那么：
- qwen-8b风格的Baseline在语义相似度评测上虚增 **10-25%**（保守估计）
- 三路方法（Baseline / RAG / Agent）的**差异被掩盖**，无法真实反映RAG和Agent的价值
- 论文的核心结论（"Agent方法优于Baseline"）可能是假阳性

**修复必要性**：必须用**原始文书文本**作为reference，或用与Baseline完全不同架构的模型（如Claude/GPT-4）做reference改写。

---

### 3.2 合规样本问题 — 没有真实合规基准

**问题描述**：

```python
# 当前方案：generate_compliance_variants-3.py
# 将违规案例改写为"假设合规"的版本
原始案例（违规）→ LLM修改关键事实 → 合规样本

# 实际问题：
# 1. 合规样本是"反事实构造"，不是真实市场行为
# 2. 真实合规商家不会用"原价XXXX元/折后XX元"的格式描述自己
# 3. LLM构造的合规场景在用词、结构上仍带有违规文书的痕迹
# 4. 200条合规样本 100% AI生成，无法验证其与真实合规行为的吻合度
```

**更好的方案**：
- 使用企业正常的商品描述、价格公示文本（如公开API数据）
- 或使用未被处罚企业的价格标示文本
- 最差情况：明确声明"合规样本为构造样本"，并在论文中限定评测范围

---

### 3.3 为什么if-else脚本难以处理 — 以PDF3 vs PDF4为例

**PDF3（犍为县）的结构噩梦**：

```
法律分析段（本局认为）：
  [正常情况] 引用法条名称+关键条款 → 100-200字
  
  [PDF3实际情况] 
  - 《价格法》第十四条 完整全文（约800字）
  - 《价格违法行为行政处罚规定》第十三条 完整全文（约600字）
  - 上述法条再次重复出现（引用+解释）
  - 总长度：3000+ 字
  
if-else脚本的失败：
  pattern = r"本局认为(.+?)依据"  
  # → 正则贪婪匹配捕获整个冗长段落，包含3次重复法条全文
  # → 无法区分"引用法条全文"和"法律分析"
  # → 提取结果：3000字法条堆砌，而非100字法律分析摘要
```

**PDF4（滁州琅琊区）的独特结构**：

```
PDF4额外包含：
  [量刑裁量段]：
    "依据《安徽省市场监督管理行政处罚自由裁量权适用规则》……
     鉴于当事人……属于从轻处罚情形……"
  
if-else脚本的失败：
  # 大多数文书没有显式量刑裁量段
  # if "从轻" in text: extract_mitigation_paragraph()
  # → PDF1/PDF2/PDF3中"从轻"只出现在处罚决定句中，没有独立段落
  # → 脚本要么漏提PDF4的裁量分析，要么错误提取其他文书
```

**根本矛盾**：

| 维度 | PDF3 | PDF4 | if-else脚本能力 |
|------|------|------|-----------------|
| "本局认为"段长度 | 3000+字（法条全文重复） | 400字（简洁） | 无法区分，只能截断或全取 |
| 量刑情节位置 | 隐含在处罚决定中 | 显式独立段落 | 需要两套完全不同的提取逻辑 |
| 证据列表格式 | 6条编号列表 | 6条编号列表（格式略不同） | 勉强可以，但边界识别脆弱 |
| 法条引用方式 | 全文引用（冗余极高） | 仅条款号引用（简洁） | 正则无法区分引用方式差异 |

**结论**：700份PDF来自不同地区、不同执法人员撰写，格式变体数量可能超过50种。纯规则脚本的维护成本将超过人工标注成本。

## 4. 可行性结论

### 4.1 预计可提取样本数量

基于4份样本的分析，对700份PDF的预估：

| 阶段 | 操作 | 预计数量 | 说明 |
|------|------|----------|------|
| 原始PDF池 | — | 700+ | 假设已全部获取 |
| Stage 1规则提取 | pdfplumber + 正则 | 650+（~93%） | 约7%因扫描件/格式损坏失败 |
| Stage 1质量筛选（≥3分） | 结构完整度评分 | 550-600（~80%） | 有完整"经查"段和处罚决定 |
| Stage 2 LLM精细结构化 | API调用 | 500-550 | LLM无法处理极端格式约5% |
| 人工审核去重后最终样本 | 人工抽样 | **450-520** | 去除重复违法行为类型 |

> **结论**：从700份PDF中构建450-600条高质量测试集是**可行的**，预计达到500条左右。

### 4.2 字段可靠性矩阵

| 字段 | 可高质量自动提取 | 需LLM辅助 | 建议人工核验 |
|------|----------------|-----------|------------|
| 文书文号、日期、机关 | ✅ | — | 抽检10% |
| 当事人名称、信用代码 | ✅ | — | 抽检10% |
| 罚款金额 | ✅ | — | 抽检10% |
| 涉及平台 | ✅（词典匹配） | — | 抽检20% |
| 违法事实原文（经查段） | ✅ | — | 不需要 |
| 法律依据条文 | ✅（正则） | — | 抽检20% |
| 违法行为类型分类 | — | ✅ | 抽检30% |
| 价格信息（原价/折扣价） | — | ✅ | 抽检30% |
| 违法摘要（50-100字） | — | ✅ | 抽检20% |
| 量刑情节 | — | ✅ | 抽检50% |
| 法律适用分析摘要 | — | ✅ | 抽检50% |

### 4.3 推荐提取策略

**不推荐**：纯if-else规则脚本（原因见第3.3节）  
**推荐**：**pdfplumber + 正则初筛 + LLM结构化**的两阶段流水线

```
Stage 1（纯规则，无LLM成本）：
  pdfplumber → 全文本 → 正则分段 → 质量评分 → 筛选合格文书
  
Stage 2（LLM辅助，有API成本）：
  合格文书 → LLM结构化提取 → JSON输出 → 入库
  
关键设计原则：
  1. Stage 1的"经查"原文直接保留，不改写（解决同源污染）
  2. LLM只做结构化，不做改写（保留原始文本）
  3. reference answer = 原始文书文本片段，而非LLM生成文本
```

## 5. 黄金测试集构建路线图

### Phase 1：批量提取 + LLM结构化（预计耗时：3-5天）

**目标**：获取500+条原始案例，每条包含结构化字段 + 原文段落

```
步骤1.1 批量规则提取
  工具：smart_pdf_extractor.py（Stage 1模式）
  输入：700+ PDF文件目录
  输出：extracted_raw.jsonl（每行一个文书的初步提取结果）
  质量评分：0-5分，筛选≥3分的文书
  预计时间：1-2小时（纯本地计算）

步骤1.2 LLM结构化精细提取  
  工具：smart_pdf_extractor.py（Stage 2模式）+ 讯飞星辰API
  输入：≥3分的文书（预计550-600份）
  输出：extracted_structured.jsonl（每行包含完整结构化字段）
  预计成本：每份文书约500-1000 tokens，总计约50万tokens
  预计时间：2-4小时（受API速率限制）

步骤1.3 字段完整度筛选
  脚本：filter_complete_records.py（见任务清单6.3）
  筛选条件：必须字段非空（经查段、处罚金额、违法类型至少有两项）
  预计输出：480-530份合格记录
```

### Phase 2：人工采样审核（预计耗时：2-3天）

**目标**：确保提取质量，发现系统性错误

```
步骤2.1 分层抽样审核
  按违法类型分层：虚构原价 / 价格不透明 / 捆绑销售 / 其他
  按平台分层：拼多多 / 美团 / 大众点评 / 淘宝 / 京东 / 其他
  抽样比例：每类至少10条，总计50-80条人工核验
  
步骤2.2 错误模式归纳
  记录最常见的提取错误类型
  更新smart_pdf_extractor.py的后处理规则
  重新处理受错误模式影响的记录

步骤2.3 多样性检查
  确保：
  - 违法行为类型多样（不要450条全是虚构原价）
  - 平台类型多样（不要全是拼多多）
  - 处罚金额分布合理
  - 地域分布广泛
```

### Phase 3：构建测试Prompt（预计耗时：2-3天）

**目标**：为每条案例构建高质量测试问题

```
关键设计决策：区分两种Query类型

Type A - Raw Evidence Query（证据查询型）：
  输入：当事人名称 + 涉及平台 + 原始证据描述
  Query示例："根据以下证据，判断是否构成价格违法行为：
              [粘贴原始经查段落的部分内容]"
  Reference：原始文书的法律分析段（本局认为）
  
Type B - Investigation Conclusion Query（调查结论型）：
  输入：已知违法事实摘要 + 要求给出处理意见
  Query示例："某电商店铺在拼多多首页标示价格49元，
              但详情页实际价格为79元，请分析是否违法。"
  Reference：相关法律依据 + 处罚决定

Type A更难，更能区分RAG和Agent的价值
Type B更接近实际用户使用场景
建议比例：Type A 40% + Type B 60%

步骤3.1 Template设计（每种Query类型2-3个模板）
步骤3.2 批量生成Query（使用提取的结构化字段填充模板）
步骤3.3 Query去重和质量检查（避免过于相似的测试样本）
```

### Phase 4：标注Reference Answer（预计耗时：3-5天）

**目标**：为每条测试样本提供高质量参考答案

```
关键原则：reference answer必须来源于原始文书，而非LLM生成

方法A（推荐）- 直接引用原文：
  Reference = 原始文书的"本局认为"段 + "处罚决定"段
  优点：零同源污染风险
  缺点：格式可能与实际query不完全对齐

方法B - 人工编写：
  由学生本人基于原文书撰写标准答案
  50-80条精标集，其余用方法A
  
方法C（慎用）- 非qwen模型改写：
  用Claude/GPT-4对原文进行格式规范化（不改变内容）
  明确标注"reference由Claude规范化"
  仍存在一定污染风险，但远低于qwen-8b

推荐方案：50条精标（方法B）+ 450条原文引用（方法A）
论文中说明：测试集中reference answer直接来源于行政处罚原文
```

## 附录：关键技术决策汇总

| 决策点 | 推荐方案 | 放弃方案 | 理由 |
|--------|----------|----------|------|
| 文本提取工具 | pdfplumber | PyPDF2、PDFMiner | pdfplumber在中文处理和布局保留上更优 |
| 结构化方法 | 正则初筛 + LLM精细 | 纯if-else规则 | PDF3/PDF4的格式差异证明纯规则不可行 |
| Reference来源 | 原始文书文本 | LLM改写文本 | 彻底消除同源污染 |
| LLM模型选择 | 讯飞星辰（结构化任务） | qwen-8b | qwen-8b = Baseline模型，不能用于生成测试集 |
| 合规样本策略 | 明确说明无真实合规样本 | LLM构造合规场景 | 论文诚信要求，避免方法论缺陷 |
| 评测指标 | 法律准确性 + BERTScore + 人工 | 纯BLEU/ROUGE | 法律任务需要内容准确性指标 |
