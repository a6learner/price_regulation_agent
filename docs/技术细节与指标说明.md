## 1. RAG 主要代码分析

### 1.1 主流程

RAG 的单样本评测主流程在 `RAGEvaluator.evaluate_single_case()` 中，核心步骤为：

- 从评测样本提取查询文本
- 执行混合检索（法规为主）
- 构建 RAG Prompt
- 调用模型并解析输出
- 计算分类、质量、性能指标

当前正式评测口径已固定为 `cases_k=0`，即不注入案例检索结果，避免同源污染风险。

### 1.2 检索模块设计

`HybridRetriever` 采用两阶段策略：

- **召回阶段**：向量语义检索 + BM25 词法检索（法规库）
- **排序阶段**：RRF 融合 + CrossEncoder 重排（可开关）

关键实现点：

- `cases_k == 0` 时直接短路，不查询 case collection
- 法规检索仍保留动态过滤：
  - 距离阈值过滤（`distance_threshold`）
  - 最小保留数（`min_k`）
  - 基于平均距离的动态 Top-K

### 1.3 当前 RAG 设计取舍

- 保留法规检索增强，降低“纯模型臆断”风险
- 暂停案例注入，降低数据同源污染对评测结论的影响
- 通过 `retrieval_info` 返回 laws/cases 数量，保证评测可追踪

## 2. Agent 主要代码分析

### 2.1 总体编排

`AgentCoordinator.process()` 采用 6 节点线性编排：

1. `IntentAnalyzer`：规则提取意图、实体、复杂度  
2. `AdaptiveRetriever`：根据意图动态设置 Top-K  
3. `Grader`：多维评分与低质量过滤  
4. `ReasoningEngine`：结构化 CoT 推理并生成 JSON  
5. `Reflector`：启发式校验 + 必要时一次重推理  
6. `RemediationAdvisor`：生成整改建议

### 2.2 各节点职责与实现特征

- **IntentAnalyzer**：纯规则实现，不额外调用 LLM，降低时延与成本；且已将 `cases_k` 固定为 0。  
- **AdaptiveRetriever**：复用 RAG 的 `HybridRetriever`，保证检索能力一致。  
- **Grader**：按相关性/覆盖度/时效性加权评分（0.6/0.3/0.1），并设置最小保留。  
- **ReasoningEngine**：将筛后的法规与案例（若有）注入系统提示词，要求输出标准 JSON 与 5 步推理链。  
- **Reflector**：先做零成本启发式校验，仅在严重问题时触发重推理；最大重试次数默认为 1。  

### 2.3 当前 Agent 状态

- 已有端到端可运行链路  
- 定位为“实验性实现，持续迭代中”  
- 主要价值在于：把检索、判定、校验与整改建议串成可解释工作流

## 3. 当前测试指标与采用原因

## 3.1 指标分层

项目当前指标分为三层：

- **分类层**：Accuracy / Precision / Recall / F1 / 混淆矩阵 / 违规类型准确率  
- **质量层**：法律依据质量分、推理质量分、两者覆盖率  
- **性能层**：平均响应时长、输入输出 Token、总 Token

### 3.2 分类层指标（为什么用）

- **Accuracy**：给出整体正确率，最直观。  
- **Precision / Recall / F1**：在违规识别任务中区分“误报”和“漏报”，比单一 Accuracy 更稳健。  
- **混淆矩阵（TP/FP/FN/TN）**：便于定位模型偏差方向。  
- **违规类型准确率**：在二分类正确前提下评估细分类型判别能力。  

> 原因：价格合规场景不仅关心“是否违规”，也关心“错在什么类型”；因此必须保留二分类与细分类双层评价。

### 3.3 质量层指标（为什么用）

质量指标由 `ResponseParser` 的启发式函数计算：

- **法律依据质量分（0-1）**  
  - 是否给出法律依据  
  - 是否提及关键法律  
  - 是否包含具体条款号  

- **推理质量分（0-1）**  
  - 是否有推理文本  
  - 是否包含事实、法律分析、逻辑连接  
  - 是否具备基本结构完整性（句子数）

> 原因：分类正确并不等于可用于合规解释。该层指标用于约束“可解释性与规范性”，弥补纯分类指标的不足。

### 3.4 性能层指标（为什么用）

- **响应时长**：衡量可用性与吞吐能力。  
- **Token 用量**：衡量成本，尤其适合比较 Baseline / RAG / Agent 的工程代价。  

> 原因：本项目是技术探索，不仅比较效果，也比较“效果-成本”平衡。

### 3.5 指标边界说明（必须明确）

- 质量分是启发式评分，不等于“法律结论事实正确率”  
- 质量分更适合做横向比较与趋势观察，不应替代人工法务审查

## 4. 为什么这套指标适合当前项目

本项目目标是比较 Baseline / RAG / Agent 三条路线。  
当前指标体系满足三类需求：

- **科研可比性**：统一输入、统一输出、统一分类指标  
- **业务可解释性**：加入法律依据与推理质量  
- **工程可落地性**：加入时延与 token 成本

因此，这是一套“效果 + 质量 + 成本”平衡的实验指标体系，适合当前阶段的技术探索与论文实验章节。

## 5. Chroma 案例库来源、与法律库关系、构建流程

### 5.1 案例库（cases collection）数据来源

当前 Chroma 的案例库并不是从 v4 评测集直接构建，而是来自历史处罚文书抽取结果：

- 上游源文件：`data/sft/processed/extracted_cases.jsonl`
- 处理脚本：`scripts/rag/rag_process_cases.py`
- 中间产物：`data/rag/cases_chunks.jsonl`

`CaseDataProcessor` 会读取每条案例，抽取并拼接两类信息作为向量文本：

- `violation_description`
- `law_references`

最终写入字段包括 `chunk_id`、`case_id`、`violation_type`、`platform`、`penalty_amount`、`content`。  
因此，当前案例库本质是“历史处罚案例检索库”（约 133 条），而不是“当前评测样本库”。

### 5.2 法律库（laws collection）数据来源

法律库来自本地法规文档目录：

- 上游目录：`data/laws/`
- 处理脚本：`scripts/rag/rag_extract_laws.py`
- 中间产物：`data/rag/laws_chunks.jsonl`

`LawDocumentExtractor` 会遍历 `中央 / 浙江 / 平台规则` 三类目录，按“第X条”切分条文后入库，形成 `law_XXXX` 形式的切片 ID。  
每条法律切片包含 `law_name`、`law_level`、`article`、`content`。

### 5.3 案例库与法律库的关系（设计层与运行层）

两者在 Chroma 中是并行的两个 collection：

- `price_regulation_laws`（法律条文库）
- `price_regulation_cases`（处罚案例库）

关系特征如下：

- **构建层**：两库共享同一套 embedding 模型（`BAAI/bge-small-zh-v1.5`）和同一 Chroma 持久化目录（`data/rag/chroma_db`），但数据来源彼此独立。
- **检索层**：`HybridRetriever` 同时支持 laws/cases 双路召回。
- **评测口径层（当前正式口径）**：RAG 与 Agent 链路都把 `cases_k` 固定为 `0`，即运行时不注入案例库，仅使用法律库检索结果，避免案例同源污染。

换言之：案例库“已构建并可检索”，但在正式评测路径中被策略性关闭；法律库是当前实际生效的检索增强来源。

### 5.4 向量库构建流程（端到端）

当前构建顺序固定为三步：

1. `python scripts/rag/rag_extract_laws.py`  
   生成 `data/rag/laws_chunks.jsonl`
2. `python scripts/rag/rag_process_cases.py`  
   生成 `data/rag/cases_chunks.jsonl`
3. `python scripts/rag/rag_build_vector_db.py`  
   读取上述两个 JSONL，编码后分别写入 Chroma 的 laws/cases collection

`rag_build_vector_db.py` 关键动作：

- 初始化 `VectorDatabase`（PersistentClient 持久化到 `data/rag/chroma_db`）
- `create_collections()` 创建/复用 laws 与 cases 两个 collection
- 用 `EmbedderModel.encode()` 批量向量化
- 分别 `add_documents()` 入库，并写入对应 metadata（laws: `law_name/law_level/article`; cases: `violation_type/platform`）
